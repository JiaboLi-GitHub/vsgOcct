# vsgocct ROADMAP

## 1. 项目定位

`vsgocct` 的长期目标不是“STEP 模型查看器”，而是一个可嵌入 CAM 软件的渲染与仿真引擎。

这个引擎需要同时覆盖以下几类能力：

- CAD/CAM 数据导入与装配语义保留
- 面向加工场景的高性能可视化
- 刀具、夹具、毛坯、机床的统一场景表达
- 刀路驱动的运动仿真
- 毛坯去除仿真
- 碰撞、过切、欠切、残料等加工验证
- 可嵌入桌面应用、批处理工具、后台服务的 SDK 形态

`vsgocct` 应该成为 CAM 软件中的“运行时核心”，而不是 UI 层或业务层本身。

## 2. 愿景

### 2.1 最终愿景

在未来，`vsgocct` 应具备以下能力：

- 读取 STEP/XCAF/OCCT B-Rep 数据并保留装配树、颜色、名称、零件语义
- 加载刀具、刀柄、夹具、毛坯、机床结构并建立统一场景
- 接收刀路数据并驱动刀具与机床轴系运动
- 在渲染线程和仿真线程之间稳定同步状态
- 完成实时或准实时毛坯去除
- 输出碰撞、干涉、过切、欠切、残料、加工包络等结果
- 支持交互式 UI 和无界面批处理两种运行模式

### 2.2 非目标

短中期内，`vsgocct` 不直接承担以下职责：

- CAM 工艺编程界面本身
- 刀路生成算法本身
- 后处理器本身
- PDM/PLM/ERP 集成
- 复杂业务工作流编排

这些应由上层系统负责，`vsgocct` 只提供标准化底层能力。

## 3. 核心原则

- 几何语义优先：不能只保留三角网格，必须保留零件、面、装配、机床部件等语义映射。
- 可验证优先：仿真不仅要“能看”，还要能输出可复核的检测结果。
- 可嵌入优先：库设计优先，示例程序只是验证载体。
- 增量演进：先做 3 轴铣削核心链路，再扩展 5 轴和更复杂工艺。
- 性能工程前置：从一开始就考虑缓存、实例复用、异步处理和 GPU 友好数据布局。
- 双模运行：同一套核心既能驱动交互式 GUI，也能跑 headless 校验。

## 4. 推荐产品路线

建议先聚焦一个明确可落地的产品面：

- 第一主线：3 轴铣削渲染与仿真
- 第二主线：3+2 定位加工
- 第三主线：5 轴联动仿真
- 第四主线：车削、车铣复合、多通道机床

原因很直接：

- 3 轴铣削的场景模型更稳定
- 毛坯去除可先采用高度场或 dexel 路线
- 验证指标更容易建立
- 更适合作为首个可商业化的引擎版本

## 5. 当前现状评估

当前仓库已经具备的基础：

- `vsgocct` 已经从示例中拆成独立库
- 已能通过 OCCT 读取 STEP，并转换为 VSG 可渲染场景
- 已有 `vsgQt` 示例作为最小查看器

当前明显缺失的核心能力：

- 没有装配树和元数据模型
- 没有零件、面、节点的稳定 ID 系统
- 没有缓存格式
- 没有刀具/毛坯/机床数据模型
- 没有刀路数据模型
- 没有仿真时序框架
- 没有毛坯去除模块
- 没有碰撞和过切验证模块
- 没有 headless 模式
- 没有系统化测试和性能基线

结论：当前代码可视为“图形导入原型”，距离“CAM 仿真引擎”还有完整的架构建设工作。

## 6. 目标架构

建议未来将 `vsgocct` 拆成如下模块。

### 6.1 `vsgocct::cad`

职责：

- STEP/XCAF/BREP/IGES 等输入
- OCCT Shape 装配树解析
- 颜色、名称、层、属性提取
- `TopoDS_Shape` 与内部语义对象映射

核心产物：

- `ShapeGraph`
- `ShapeNode`
- `ShapeId`
- `FaceId`
- `AssemblyNodeId`

### 6.2 `vsgocct::mesh`

职责：

- OCCT 三角化参数控制
- 网格生成与法线生成
- 曲面边界与锐边提取
- 多精度网格缓存

核心产物：

- `MeshOptions`
- `MeshAsset`
- `MeshCacheKey`

### 6.3 `vsgocct::scene`

职责：

- `ShapeGraph` 到 VSG 节点的构建
- 零件/面/边语义和渲染节点绑定
- 选择、高亮、隐藏、隔离等场景操作

核心产物：

- `SceneBuilder`
- `SceneHandle`
- `SceneIndex`
- `RenderableId`

### 6.4 `vsgocct::machine`

职责：

- 机床结构表达
- 坐标系链和运动学链
- 轴限位、回转轴、联动姿态求解

核心产物：

- `MachineDefinition`
- `KinematicChain`
- `AxisState`
- `MachinePose`

### 6.5 `vsgocct::tooling`

职责：

- 刀具、刀柄、接杆、夹头模型表达
- 刀具轮廓和包络表达
- 刀库与参数定义

核心产物：

- `ToolDefinition`
- `HolderDefinition`
- `ToolAssembly`

### 6.6 `vsgocct::toolpath`

职责：

- 刀位点、进给、主轴、冷却、工步等刀路数据表达
- 插补结果缓存
- 时间轴采样

核心产物：

- `ToolpathProgram`
- `Operation`
- `ToolPoseSample`
- `SimulationTimeline`

### 6.7 `vsgocct::stock`

职责：

- 毛坯表达
- 毛坯去除仿真
- 加工中间状态缓存

可选内部实现路线：

- `HeightField`
- `Dexel`
- `Voxel`
- `Signed Distance Field`

### 6.8 `vsgocct::verify`

职责：

- 碰撞检测
- 干涉分类
- 过切/欠切/残料分析
- 加工结果报告

核心产物：

- `CollisionReport`
- `VerificationResult`
- `StockComparisonResult`

### 6.9 `vsgocct::runtime`

职责：

- 仿真任务调度
- 后台线程和前台渲染线程协调
- 时间推进与状态同步
- GUI 与 headless 统一入口

## 7. 里程碑总览

## Milestone M0: 库化基础稳定

目标：

- 把当前原型正式变成可扩展库

交付：

- 清晰的 `include/`、`src/`、`examples/` 目录结构
- 导出的 `vsgocct` 库目标
- `find_package(vsgocct)` 的安装与导出能力
- API 文档和最小示例

完成标准：

- 第三方项目可以单独链接 `vsgocct`
- 示例程序不再直接依赖内部源码文件
- 代码中“读取、三角化、建场景”完成模块拆分

状态：

- 已部分启动

## Milestone M1: CAD 语义层

目标：

- 从“读出网格”升级到“读出带语义的装配模型”

交付：

- `ShapeGraph`
- 装配树
- 零件名称、颜色、层、可见性
- 稳定 `ShapeId/FaceId/NodeId`
- XCAF 导入链路

完成标准：

- 同一个零件在场景、装配树、属性面板之间可稳定关联
- 选择一个可渲染节点能反查到原始 `TopoDS_Shape`
- 支持按装配层级显示/隐藏

## Milestone M2: 可用的 CAD 查看器内核

目标：

- 让 `vsgocct` 成为一个合格的 CAD/CAM 查看底层

交付：

- 选择、高亮、隔离、隐藏、透明显示
- 边线显示、轮廓显示
- 剖切平面
- 视图包围盒定位
- 颜色和材质管理

完成标准：

- 能支撑上层 UI 搭建一个实用的装配查看器
- 单模型和中等装配都可流畅交互

## Milestone M3: CAM 场景模型

目标：

- 定义“加工场景”的统一对象模型

交付：

- 工件、目标件、毛坯、夹具、刀具、刀柄、机床模型
- 场景节点分类和统一状态管理
- 坐标系和变换约定

完成标准：

- 可以在同一场景中同时表达零件、毛坯、刀具和机床
- 支持基础场景保存与恢复

## Milestone M4: 刀路驱动与时序框架

目标：

- 从静态查看器进入“加工仿真播放”阶段

交付：

- 刀路数据结构
- 工步和程序层次
- 时间轴推进器
- 刀具姿态插值
- 机床姿态同步

完成标准：

- 给定一段刀路，能稳定播放刀具和机床运动
- 支持暂停、继续、跳转、单步、倍速

## Milestone M5: 3 轴毛坯去除原型

目标：

- 做出第一版可用的减材仿真能力

建议技术路线：

- 优先采用 `heightfield` 或 `dexel`
- 不追求第一版就做全局精确布尔

交付：

- 矩形毛坯表达
- 3 轴铣削去除
- 实时或准实时更新
- 毛坯网格可视化

完成标准：

- 典型 3 轴示例可连续播放
- 毛坯变化与刀具轨迹基本一致
- 性能达到可交互级别

## Milestone M6: 碰撞与安全校验

目标：

- 支撑 CAM 仿真最核心的“有没有问题”判断

交付：

- 刀具与工件碰撞

- 仿真过程中能产生清晰的碰撞事件

## Milestone M7: 加工结果验证

目标：

- 从“过程仿真”进入“结果验证”

## 9. 技术路线建议

### 9.1 几何语义

必须尽快引入以下内部对象，而不是继续只返回 `vsg::Node`：

- `ShapeGraph`
- `ShapeInstance`
- `SceneEntity`
- `SelectionToken`
- `SemanticMetadata`

没有这一层，后续所有 CAM 能力都会变得很脆。

### 9.2 毛坯仿真

建议不要一开始用 OCCT 精确布尔做实时减材主路线。

原因：

- 精确布尔稳定性和性能都不适合作为逐帧主路径
- 3 轴连续切削更适合 `heightfield/dexel`
- 精确布尔更适合做局部复核或离线验证

建议策略：

- 3 轴：`heightfield` 或 `dexel`
- 3+2：先沿用 `dexel`
- 5 轴：`voxel/SDF` 或混合方案
- 精确验证：局部 swept volume + OCCT 复核

### 9.3 碰撞检测

建议使用两级检测：

- 粗检测：包围盒、层级包围体、空间索引
- 精检测：刀具轮廓、三角网格或解析包络

不要直接把所有碰撞都落到精确 B-Rep 级别。

### 9.4 渲染架构

渲染层要从一开始支持：

- 实例化
- 大装配分层显示
- 延迟更新
- 异步几何提交
- 面/边/选中覆盖渲染
- 剖切与透明件

### 9.5 数据缓存

建议尽早定义缓存：

- 语义缓存
- 网格缓存
- 毛坯中间状态缓存
- 仿真关键帧缓存

否则每次启动都重新读 STEP、重新三角化、重新构场景，会严重拖慢整个系统。

## 10. API 演进建议

未来建议从单一函数：

`loadStepScene(path)`

演进到类似下面的分层 API：

```cpp
auto shapeGraph = vsgocct::cad::readStep(path, readerOptions);
auto meshSet = vsgocct::mesh::buildMeshes(shapeGraph, meshOptions);
auto scene = vsgocct::scene::buildScene(shapeGraph, meshSet, sceneOptions);

auto machine = vsgocct::machine::loadDefinition(machineFile);
auto toolpath = vsgocct::toolpath::load(programFile);
auto session = vsgocct::runtime::createSession(scene, machine, toolpath, stockOptions);
session->play();
```

这个方向更适合 CAM 引擎，而不是把所有职责都塞进一个 loader。

## 11. 性能目标

需要尽早建立量化目标。

建议至少定义以下指标：

- STEP 导入耗时
- 三角化耗时
- 场景构建耗时
- 首帧时间
- 交互帧率
- 刀路回放帧率
- 毛坯更新耗时
- 碰撞检测耗时
- 内存峰值

建议建立三档样例：

- 小模型
- 中型装配
- 大型装配

每个版本都跑一次基准。

## 12. 测试策略

至少需要四类测试：

### 12.1 导入测试

- STEP/XCAF 样例是否能正确导入
- 装配层级、颜色、名称是否完整

### 12.2 几何测试

- 网格数量、三角数、法线、包围盒是否稳定

### 12.3 仿真测试

- 刀路播放结果是否稳定
- 关键帧姿态是否正确
- 毛坯去除结果是否符合预期

### 12.4 验证测试

- 碰撞检测是否能命中标准案例
- 过切/欠切分析是否符合基准样例

## 13. 风险与应对

### 风险 1：一开始目标过大

问题：

- 同时做 5 轴、车削、复合机床、精确去除会失控

应对：

- 先锁定 3 轴铣削

### 风险 2：几何语义丢失

问题：

- 如果只保留三角网格，后续所有选择、报告、验证都会很难做

应对：

- 第一阶段就建立 `ShapeGraph + ID 映射`

### 风险 3：实时精确布尔性能不足

问题：

- OCCT 精确布尔不适合直接做实时主链路

应对：

- 主路径采用 `dexel/voxel/SDF`
- 精确布尔只做复核

### 风险 4：渲染与仿真线程耦合

问题：

- 容易造成卡顿、状态错乱和难以排查的问题

应对：

- 早期就建立 runtime 调度与双线程数据交换模型

### 风险 5：没有样例基准

问题：

- 项目容易越做越复杂，但无法量化进展

应对：

- 建样例库、基准集和回归集

## 14. 组织方式建议

建议把项目管理拆成 5 条长期工作流：

- 几何与数据导入
- 渲染与场景管理
- 机床与刀路仿真
- 毛坯与验证
- 工程化与性能

每条工作流都应有自己的 backlog 和验收标准。

- 提供 GUI 示例和 headless 接口
- 提供稳定公共 API

## 17. 结论

`vsgocct` 如果要成为 CAM 软件的渲染仿真引擎，核心不在“再多支持几个文件格式”，而在于完成三次升级：

- 从网格导入器升级为 CAD 语义引擎
- 从查看器升级为加工场景 runtime
- 从动画播放器升级为加工验证内核

短期应该把基础库架构做正确。
中期应该把 3 轴加工场景和去除仿真做出来。
长期再扩展到 5 轴、复杂机床和工业级验证。
